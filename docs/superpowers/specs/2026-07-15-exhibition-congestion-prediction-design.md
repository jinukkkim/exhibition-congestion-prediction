# 실시간 전시 혼잡도 예측 시스템 — MVP 설계

- 날짜: 2026-07-15
- 목적: 취업 포트폴리오용 개인 프로젝트
- 타임라인: 8주

## 1. 목표 & 범위

**무엇을 만드나:** 국립중앙박물관 하나에 대한 실시간 혼잡도 표시 + 미래 혼잡도 예측 웹 서비스.

국립중앙박물관은 서울시 실시간 도시데이터 API가 지원하는 주요 120(121)장소 목록에 포함되어 있음 (매뉴얼 "국립중앙박물관·용산가족공원" 항목으로 확인, [서울 열린데이터광장](https://data.seoul.go.kr/dataVisual/seoul/guide.do)).

**차별점:** 서울시 API는 "현재" 혼잡도만 제공한다. 이 프로젝트는 그 위에 **"N시간 뒤 혼잡도" 예측**을 얹어 차별화한다. 단순히 API 값을 재표시하는 것과 구분되는 핵심 가치.

### 포함 (In Scope)
- 서울시 실시간 도시데이터 API 1개 소스, 5분 주기 수집 → DB 적재 (설계 승인 시점부터 8주간 계속 축적)
- 축적된 데이터 기반 요일×시간대 통계 베이스라인 예측
- 동일 데이터로 학습한 scikit-learn 모델(예: GradientBoosting)과 베이스라인의 정확도(MAE) 비교
- 실시간 현재 혼잡도 표시 (새로고침 없이 갱신, SSE)
- 오늘 하루 시간대별 예측 혼잡도 곡선 표시

### 제외 (Out of Scope, YAGNI)
- 다른 장소로 확장 (국립중앙박물관 하드코딩. 구조상 확장 가능하되 지금 구현하지 않음)
- 날씨·네이버 데이터랩·SNS 해시태그 등 추가 데이터 소스
- 회원가입/알림/즐겨찾기 등 부가 기능
- 프로덕션 배포 인프라 (필요 시 별도 논의)

## 2. 기술 스택

기존 `chatbot-project`의 스택을 재사용한다 (이미 익숙하고, 이 프로젝트 규모에 적절한 무게).

- **백엔드**: FastAPI (Python), SQLAlchemy + Postgres, Redis
- **프론트엔드**: React + Vite + TypeScript + Tailwind
- **예측 모델**: scikit-learn

`SKKUDING/codedang` (NestJS + GraphQL + Prisma + BullMQ + OpenTelemetry + Next.js)은 팀 규모 마이크로서비스 스택이라 8주 솔로 프로젝트엔 과도해 채택하지 않는다.

## 3. 아키텍처

```
[APScheduler] --5분마다--> [서울시 실시간 도시데이터 API]
        |
        v
  [수집 워커] --적재--> [Postgres: raw_congestion 테이블]
        |
        v
  [Redis] <-- 최신 값 캐싱 (조회용, TTL 5분)
        |
        v
  [FastAPI 백엔드] --SSE--> [React 프론트엔드]
        ^
        |
  [배치 작업] --하루 1회--> 베이스라인 통계 갱신 + ML 모델 재학습
```

- **수집기**: FastAPI 앱 내 인프로세스 스케줄러(APScheduler)가 5분마다 서울시 API를 호출해 Postgres에 원본 적재. 호출 빈도가 낮아(1개 API, 5분 주기) Celery 같은 별도 작업 큐는 불필요 — 오버엔지니어링 방지.
- **캐시**: 최신 혼잡도 값과 오늘의 예측 곡선을 Redis에 캐싱. 다수 사용자 동시 조회 시에도 DB 부하 없이 응답.
- **실시간 통신**: WebSocket이 아닌 **SSE**를 사용한다. 서버→클라이언트 단방향 브로드캐스트만 필요하고(클라이언트가 서버로 보낼 메시지 없음), SSE는 일반 HTTP 위에서 동작해 자동 재연결이 기본 제공된다.
- **예측**: 하루 1회 배치 작업이 (a) 요일×시간대 평균 베이스라인을 갱신하고, (b) scikit-learn 모델을 재학습한다. 결과는 Redis에 캐싱해 프론트에서 즉시 조회.

## 4. 데이터 흐름 & 폴백

- **API 호출 실패** (타임아웃, 서울시 서버 장애): 직전 성공 값을 Redis에 유지하고 "마지막 갱신 시각"을 프론트에 표시. 별도 재시도 큐 없이 다음 5분 주기에 자연스럽게 재시도.
- **콜드 스타트**: 예측 모델 학습에 필요한 최소 데이터(예: 2주치)가 쌓이기 전에는 베이스라인/ML 예측 대신 "데이터 수집 중" 상태를 노출한다.

## 5. 테스트

- **수집기**: 서울시 API 응답을 mock으로 주입해 파싱/저장 로직 단위 테스트.
- **예측 모델**: 베이스라인 vs ML의 MAE(평균 절대 오차)를 비교하는 스크립트. 이 비교 결과 자체가 포트폴리오에서 보여줄 모델 검증 근거가 된다.
- **프론트**: SSE로 값이 갱신될 때 화면이 갱신되는지 확인하는 e2e 테스트 1개.

## 6. 의사결정 요약

| 항목 | 결정 | 이유 |
|---|---|---|
| 장소 범위 | 국립중앙박물관 1곳 | 포트폴리오 마감(8주) 내 완성도 우선, 확장 여지는 구조로만 남김 |
| 데이터 소스 | 서울시 실시간 도시데이터 API 1개 | 소스 추가마다 연동/가중치 튜닝 비용 증가, MVP엔 불필요 |
| 차별화 포인트 | 미래 예측 | 서울시 API가 이미 현재 혼잡도를 제공하므로, 단순 재표시가 아닌 예측으로 가치 추가 |
| 콜드 스타트 대응 | 지금부터 수집 시작, 파이프라인/백엔드/프론트와 병행 개발 | 5~7주치 데이터로 예측 모델 학습·검증 가능한 최소 조건 확보 |
| 예측 모델 | 통계 베이스라인 + scikit-learn 모델 비교 | 적은 데이터로도 검증 가능한 베이스라인 확보 + 모델 평가 근거로 포트폴리오 설득력 강화 |
| 실시간 통신 | SSE (WebSocket 아님) | 단방향 브로드캐스트만 필요, 구현/운영이 더 단순 |
| 기술 스택 | FastAPI + Postgres + Redis + React/Vite/TS | 기존 chatbot-project 스택 재사용, 프로젝트 규모에 적합 |
